# 日志、测试与代码审查规范

## Zap 日志规则

- 使用结构化字段记录 `requestId`、`userId`、`path`、`method`、`latency`。
- 错误日志必须带错误对象与业务上下文。
- 区分 `Info`、`Warn`、`Error`，不要把所有异常都打成 Error。
- 访问日志与业务日志分工明确，避免重复打印。

## 测试规则

- Service 层至少覆盖成功、失败、边界分支。
- Handler 测试覆盖绑定失败、业务错误、成功响应。
- Repository 测试优先使用隔离数据库、测试容器或事务回滚策略。
- 对中间件测试请求 ID 透传、鉴权失败、限流触发、panic recovery 等关键路径。

## 代码审查清单

- [ ] 路由与 HTTP 方法语义一致。
- [ ] Handler 无业务逻辑。
- [ ] 响应体与错误码统一。
- [ ] 参数校验完整且错误信息稳定。
- [ ] 日志无敏感信息。
- [ ] 数据库查询具备分页、过滤或边界控制。
- [ ] 事务边界合理，无长耗时外部调用。
- [ ] 关键业务分支存在测试。
- [ ] 配置、JWT 密钥、数据库连接信息未硬编码。

## 推荐实践

- 为统一响应、错误码转换、中间件链顺序建立快照测试或约束测试。
- 对关键模块补充 review checklist，作为 PR 模板的一部分。
- 代码审查优先关注行为一致性、分层边界、异常路径和安全风险。

## 禁止事项

- 禁止：只测 happy path。
- 禁止：用日志断言替代真实行为断言。
- 禁止：提交前不检查统一返回体、错误码和中间件顺序。
