# 错误处理规范

## 错误分类

- 参数错误
- 鉴权错误
- 权限错误
- 资源不存在
- 业务冲突
- 系统内部错误

## 错误码策略

- 使用统一错误码而不是散乱的字符串消息。
- 错误码按模块分段，例如用户、订单、支付分别保留区间。
- 对外错误消息保持稳定、可理解，对内日志记录完整上下文。

## 推荐模式

```go
var ErrUserNotFound = bizerr.New(40401, "user not found")
var ErrEmailExists = bizerr.New(40901, "email already exists")
```

## 强制规则

- Logic 返回业务错误，Handler 或统一响应层负责映射 HTTP 状态与返回体。
- 未知错误统一降级为内部错误并记录结构化日志。
- 包装错误时保留原始错误链，便于 `errors.Is`、`errors.As` 判断。

## 推荐实践

- 将“错误定义”和“错误到响应的映射”分开管理。
- 对外部依赖错误进行归类，例如数据库超时、下游超时、熔断、限流。
- 对重复资源、状态冲突、幂等冲突使用明确业务错误，而不是一律 500。

## 禁止事项

- 禁止：直接把 `err.Error()` 返回给前端。
- 禁止：在每个 Handler 复制粘贴错误输出逻辑。
- 禁止：混用 HTTP 状态码与业务错误码导致语义不清。
