# 中间件规范

## 推荐顺序

1. Recovery
2. Request ID
3. Access Log
4. CORS
5. Auth
6. Rate Limit

## 规则

- Recovery 应尽早注册，统一拦截 panic 并输出标准错误响应。
- Request ID 应在请求早期注入上下文与响应头，后续日志统一复用。
- Access Log 应记录方法、路径、状态码、耗时、请求 ID、调用方标识。
- Auth 中间件只负责身份鉴别，不承载复杂业务授权。
- Rate Limit 规则要按路径、租户、用户或 IP 明确维度，不能混乱叠加。

## 推荐实践

- 使用类型安全的上下文 key，避免字符串 key 冲突。
- 将跨域配置、黑白名单、限流阈值全部配置化。
- 对需要跳过鉴权的健康检查、登录接口使用显式路由分组，而非中间件内部写大量例外。

## 禁止事项

- 禁止：在中间件中编排业务逻辑。
- 禁止：中间件之间通过临时字符串 key 随意传数据。
- 禁止：记录完整敏感请求体、密码、Token、密钥。
- 禁止：让鉴权中间件直接操作数据库事务。
