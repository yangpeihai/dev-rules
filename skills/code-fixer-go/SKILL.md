# Go 代码修复助手

帮助 AI 或开发者根据检查报告、问题清单或自检结果，对 Go 代码执行最小必要修复，并输出可追踪、可验证、可继续迭代的修复记录。

## Overview

这个 skill 的重点不是“尽量多改”，而是“把明确问题逐项修掉，并且知道哪些修了、哪些没修、哪些还要验证”。

它默认服务于两类场景：
- 单轮定向修复：用户给出明确问题，直接做最小补丁
- 自纠流水线：消费 `code-checker-go` 产出的 findings，逐项修复并回填状态

## AI 速用入口

如果只看这一节，也应按下面执行：

1. 先确认问题来源和允许修改范围
2. 把输入归一化成待修问题项，不要边看边随手扩散
3. 判断项目类型，只加载最相关的规则
4. 按严重程度逐项修，不要一次改一大片
5. 每修完一项就记录状态：`fixed`、`blocked`、`skipped`、`needs_verification`
6. 每个标记为 `fixed` 的问题都要给出最小验证依据

### 最短输出模板

```markdown
## Fix Result
- [GO-H001] fixed
  修改：
  验证：

- [GO-M001] blocked
  原因：

## Summary
- 已修复：
- 需验证：
- 阻塞：
```

### 速停条件

出现以下情况时，不要继续硬修：
- 问题无法安全定位
- 最小修复会牵动大范围重构
- 缺少关键上下文或测试入口
- 规则冲突会直接影响修复落点

## AI 触发条件

当任务满足以下任一条件时，应优先使用本规范：
- 根据 `code-checker-go` 报告修复 Go 代码
- 根据用户给出的明确 issue 列表做定向修复
- 在 AI 自检后继续自纠，形成检查到修复的闭环
- 需要最小范围改动并输出结构化修复结果

## AI 决策顺序

按以下顺序处理，避免一边修一边扩散：

1. 先确认修复来源：用户指令、review 报告、issue 列表、自检结果
2. 再确认允许修改的范围
3. 再把输入归一化为“待修问题项”
4. 再判断项目类型和适用规则
5. 逐项决定：修复、跳过、阻塞，还是先验证
6. 每修完一项就做最小验证并回填结果

## 核心边界

### 必须遵守

- 只修复用户明确授权的问题及其直接必要上下文
- 保持修改最小化，不把修复升级成顺手重构
- 每个修复项都要能追溯到问题来源
- 无法安全修复的项必须明确标记，而不是静默跳过

### 不应做的事

- 不做“顺手优化”“顺手统一风格”“顺手大改结构”
- 不因为发现了更多问题就自动扩大修复范围
- 不为了好看而重写整个文件，除非这是修复所必需
- 不宣称“已修复”却没有对应证据或最小验证

## 修复范围确定

在开始修改前，必须先明确本轮允许修改的范围，不得因为顺手发现其他问题而自动扩散。

### 范围优先级

1. 用户显式授权的修改范围
2. 原始问题项直接涉及的文件、函数或模块
3. 为完成最小修复所必需的直接上下文
4. 仅在用户明确要求时，才扩大为模块级或全盘修复

### 常见范围类型

| 范围类型 | 说明 | 适用场景 |
|------|------|------|
| 本次变更修复 | 只修当前 diff 或当前任务直接涉及的问题 | 提交前修复、PR review 回应 |
| 定向问题修复 | 只修指定 issue、指定文件或指定模块中的明确问题 | review findings 回填、专项修复 |
| 最小关联扩展修复 | 为闭环修复问题而补改紧邻调用链、接口或测试 | 错误处理补齐、分层职责回收 |
| 全盘治理修复 | 跨模块、跨目录统一修复同类问题 | 基线治理、统一整改、专项清理 |

### 默认策略

- 用户未明确说明时，默认按“定向问题修复”或“本次变更修复”处理
- 只有当原问题无法在当前范围内安全闭环时，才扩展到最小必要上下文
- 扩展修改范围时，应明确说明为什么必须扩展、实际改到了哪些位置
- 未经用户明确授权，不默认进行全盘治理或顺手重构

## 输入归一化

支持以下输入来源，按优先级处理：

1. 用户本轮显式指定的问题
2. `code-checker-go` 生成的 findings 报告
3. 用户提供的 issue 列表、日志、报错信息

无论来源如何，都先归一化为统一问题项：

| 字段 | 说明 |
|------|------|
| `issue_id` | 原始问题编号，若无则生成临时编号 |
| `severity` | 严重级别 |
| `category` | 问题类别 |
| `location` | 文件、结构位置、行号 |
| `summary` | 问题摘要 |
| `basis` | 问题依据 |
| `suggested_fix` | 原始修复方向 |
| `status` | 当前修复状态 |

如果输入字段不全，应先补足理解，再进行最保守修复。

## 规则路由

默认按以下顺序选择规则：

1. 用户本轮显式要求
2. 当前项目中的规则文档、skills、团队约定
3. `go-rule`
4. `gin-web-rule` 或 `go-zero-rule`
5. Go 通用惯例与标准工具链

### 路由原则

- 通用问题，如错误处理、资源释放、`context`、并发安全，默认由 `go-rule` 兜底
- Gin 分层、统一返回体、GORM 事务边界等问题，优先联动 `gin-web-rule`
- go-zero 的 `handler/server -> logic -> model`、`ServiceContext`、错误码映射等问题，优先联动 `go-zero-rule`
- 同一问题同时涉及框架边界和通用 Go 写法时，先定职责落点，再修具体写法

## 修复状态机

推荐统一使用以下状态：

| 状态 | 含义 |
|------|------|
| `pending` | 待处理 |
| `fixed` | 已修复并完成最小验证 |
| `skipped` | 明确不处理，通常因不在范围内 |
| `blocked` | 缺少关键信息或无法安全修改 |
| `needs_verification` | 已做代码修改，但仍需进一步运行或联调验证 |

### 状态判定要求

- 只有完成代码修改且做过最小验证，才可标记为 `fixed`
- 已修改但无法充分验证时，标记为 `needs_verification`
- 问题描述不足、定位不清、改动风险过大时，标记为 `blocked`
- 用户未授权或本轮不处理的项，标记为 `skipped`

## 修复顺序

默认按以下顺序处理：

1. 安全问题
2. 行为错误和错误处理问题
3. 事务、资源释放、并发风险
4. 框架分层和架构边界问题
5. 测试缺口与可验证性问题
6. 一致性问题

如果低优先级问题会干扰高优先级修复，应先暂停低优先级项。

## 逐项修复闭环

对每个问题项都按以下顺序处理：

1. 识别原始问题和依据
2. 确定最小必要改动范围
3. 选择对应规则和正确落点
4. 实施最小补丁修复
5. 做最小验证
6. 回填修复状态与说明

不要跳过第 5 步和第 6 步。

## 最小验证原则

修复后至少给出一项验证依据：

- 读取相关代码，确认问题路径已被覆盖
- 运行最相关的测试或命令
- 说明为什么当前只能做到静态验证

如果无法验证，应明确写出：
- 没验证什么
- 为什么没法验证
- 用户下一步该怎么验证

## 推荐修复结果字段

为了兼容单轮输出和流水线回填，推荐每个修复项尽量包含以下字段：

| 字段 | 说明 |
|------|------|
| `issue_id` | 对应原始问题编号 |
| `status` | `fixed` / `skipped` / `blocked` / `needs_verification` |
| `file` | 主要修改文件 |
| `summary` | 一句话说明做了什么 |
| `basis` | 修复依据 |
| `verification` | 验证方式或未验证说明 |
| `notes` | 兼容性、残余风险或阻塞说明 |

## 默认评分规则

`code-fixer-go` 默认应给出评分，但这个评分不是“整个项目质量总分”，而是“本轮修复涉及的目标范围内代码质量评分”。

### 评分边界

- 只评本轮修复涉及的目标范围
- 不擅自扩大到整个项目、整个仓库或无关模块
- 评分应结合修复结果、验证情况、残余风险综合判断
- 评分必须和本轮实际证据一致，不能为了好看虚高

### 默认输出项

默认应包含以下评分字段：

- `score`：0-100 分
- `grade`：等级，如 `A` / `B` / `C` / `D`
- `scoring_basis`：评分依据，说明为什么得到这个分数

### 评分参考

评分时可参考以下因素：

- 已修复问题的严重程度和数量
- 是否仍存在 `blocked` 或 `needs_verification`
- 关键问题是否已经闭环验证
- 目标范围内是否仍有明显残余风险
- 修复是否保持了项目原有结构、风格和边界

### 评分表达要求

- 评分必须在修复结果总结中明确给出
- 必须同时给出分数、等级和评分依据
- 如果证据不足，应下调分数并在 `scoring_basis` 中明确说明
- 不得把“未验证”包装成“高质量”

## 双模式输出

### 单轮修复模式

适用场景：
- 用户给了少量明确问题
- 本轮目标是快速改完并说明结果

输出要求：
- 优先列出已修复、未修复、阻塞
- 每项说明对应问题来源和验证情况
- 默认同步生成 Markdown 文档，并在可行时同步生成对应 JSON 文件
- 默认给出目标范围代码质量评分：`score`、`grade`、`scoring_basis`

### 流水线回填模式

适用场景：
- 输入来自 `code-checker-go` 报告
- 需要把结果继续交给人或下一轮 AI

输出要求：
- 保留原始 `issue_id`
- 明确状态、摘要和验证方式
- 区分已修、需验证、阻塞、跳过
- 默认落盘为 JSON，并同时补一份 Markdown 摘要
- 默认给出目标范围代码质量评分：`score`、`grade`、`scoring_basis`

## 输出落盘规则

默认应将修复结果保存为文档，而不是只停留在对话输出中。

### 默认目录

- 默认输出目录：`docs/code-fix/`
- 如果目录不存在，应先创建目录，再写入结果文件
- 除非用户明确指定其他位置，否则不要把修复结果直接散落到 `docs/` 根目录

### 默认文件格式

- 默认同时输出两份文件：
- 一份 Markdown：给人快速阅读和 review
- 一份 JSON：给后续 AI、脚本或复测流程继续消费

### 文件命名规则

默认文件名应包含以下信息：

1. 日期
2. 时间
3. 用户本轮任务主题
4. 从当前分支名提取的简短特性名

推荐格式：

- Markdown：`YYYY-MM-DD-HHMMSS-<task-topic>-<branch-feature>.md`
- JSON：`YYYY-MM-DD-HHMMSS-<task-topic>-<branch-feature>.json`

### 命名生成原则

- `task-topic` 优先使用用户本轮任务主题，转成简短、可读、短横线连接的英文或拼音标识
- `branch-feature` 作为补充信息，从当前分支名中提取更短的特性名
- 如果分支名本身已经较短且清晰，可直接使用
- 如果分支名包含噪音前缀、用户名、流水号等信息，应只保留最能表达特性的那一段

### 落盘要求

- 对话中给出修复摘要时，也应同步说明写入了哪些文件
- Markdown 与 JSON 的核心结论必须保持一致
- 文件中必须包含问题来源、修复范围、修复结果、验证情况和残余风险

## 推荐输出结构

```json
{
  "metadata": {
    "language": "go",
    "task_topic": "fix-user-create-logic",
    "branch_feature": "http-api-tester",
    "source": "code-checker-go findings",
    "project_type": "gin-web",
    "total_issues": 3,
    "fixed": 1,
    "needs_verification": 1,
    "blocked": 1,
    "score": 82,
    "grade": "B",
    "scoring_basis": "高优先级问题已完成修复，但仍存在 1 项需进一步验证和 1 项阻塞，目标范围内仍有残余风险"
  },

    {
      "issue_id": "GO-H001",
      "status": "fixed",
      "file": "internal/logic/user_logic.go",

      "verification": "已检查调用链并运行相关测试"
    }
  ]
}
```

## 消费 `code-checker-go` findings 示例

当输入来自 `code-checker-go` 时，先复用原始 `issue_id`、`basis` 和 `suggested_fix`，再决定最小补丁形态，不重新发明问题定义。

输入问题项示例：

```json
{
  "issue_id": "GO-H001",
  "severity": "high",
  "category": "layering",
  "location": "internal/handler/user/createuserhandler.go:28",
  "summary": "Handler 中直接编排数据库写入，绕过 logic 层",
  "basis": "go-zero-rule 要求 handler/server -> logic -> model 分层",
  "suggested_fix": "将业务编排迁移到对应 logic，由 handler 仅做请求解析和响应输出",
  "status": "confirmed"
}
```

对应的修复回填结果可以是：

```json
{
  "issue_id": "GO-H001",
  "status": "fixed",
  "file": "internal/handler/user/createuserhandler.go",
  "summary": "将业务编排迁移到 CreateUserLogic，handler 仅保留请求解析与响应输出",
  "basis": "沿用原始 findings 的分层依据，按 go-zero-rule 修复职责落点",
  "verification": "已检查 handler 不再直接操作 model，并确认调用链改为 handler -> logic",
  "notes": "若项目存在配套单测，下一步应补跑相关 handler/logic 测试"
}
```

重点是保持问题语义连续，而不是把修复结果写成一段脱离原始 issue 的描述。

## AI 默认执行步骤

1. 读取问题来源并归一化为待修问题项
2. 判断项目类型，只加载最相关的 1-2 套规则
3. 按严重程度逐项修复，不扩大范围
4. 每修完一项就记录状态和验证结果
5. 总结时优先说明已修、未修、阻塞和残余风险
6. 基于目标范围内代码质量给出 `score`、`grade`、`scoring_basis`
7. 将结果同时整理为 Markdown 和 JSON，并默认写入 `docs/code-fix/`

## AI 自检清单

输出前至少快速检查：

- [ ] 是否只修了用户授权的问题和必要上下文
- [ ] 是否保留了项目已有命名、分层、事务和错误风格
- [ ] 是否没有把修复偷偷扩展成重构
- [ ] 是否每个问题项都有明确状态
- [ ] 是否每个标记为 `fixed` 的项都有最小验证依据
- [ ] 是否明确写出了 `blocked` 或 `needs_verification` 的原因
- [ ] 是否已给出目标范围代码质量评分
- [ ] 是否同时包含 `score`、`grade`、`scoring_basis`
- [ ] 是否评分与验证证据、残余风险相匹配
- [ ] 是否已明确说明哪些最小验证已执行、哪些未执行及原因
- [ ] 是否已将结果写入默认文档目录
- [ ] 是否文件名包含日期时间、任务主题和分支特性名
- [ ] 是否同时生成了 Markdown 和 JSON 两份结果

## 停止条件

出现以下情况时，应暂停继续修改并明确说明：

- 输入问题无法安全定位
- 最小修复不可避免牵动大范围重构
- 规则冲突会直接影响修复落点
- 缺少运行环境、测试入口或关键上下文，无法继续安全推进

## AI 默认输出要求

- 优先列出“已修复 / 未修复 / 阻塞 / 需验证”
- 每项尽量包含来源、位置、处理结果、验证方式
- 若输入来自检查报告，优先保留原始 `issue_id`
- 没做验证时必须明确说明，而不是省略
- 默认给出目标范围代码质量评分：`score`、`grade`、`scoring_basis`
- 默认将结果写入 `docs/code-fix/`

## 何时联动其他文档

- **按报告做通用 Go 修复** → `code-checker-go` + `go-rule`
- **修 Gin Web 分层、事务、返回体问题** → `gin-web-rule` + `go-rule`
- **修 go-zero 分层、错误码、ServiceContext 问题** → `go-zero-rule` + `go-rule`
- **只做最小补丁修复** → 当前 skill + 原始问题来源

## 常见误区

| 误区 | 正确做法 |
|------|------|
| 看到别的问题就顺手一起改 | 只修明确问题及其必要上下文 |
| 改完代码就直接宣称修复完成 | 至少说明做了什么最小验证 |
| 把无法确认的问题硬修掉 | 标记 `blocked` 或 `needs_verification` |
| 为了统一风格大面积重写 | 优先最小补丁和项目一致性 |

## 最终目标

这个 skill 的产出应当满足两件事：

1. 人可以快速看出哪些问题已经被安全处理
2. AI 可以把修复结果继续回填到下一轮检查或交付说明里
