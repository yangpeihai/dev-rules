# 代码审查清单

## 核心检查点

- 先看分层边界是否被破坏，再看具体实现细节。
- 优先关注错误路径、事务边界、日志上下文和生成代码改动风险。

## 提交前检查

- [ ] 代码已通过 `gofmt` 和 `goimports` 格式化
- [ ] Handler 层仅做请求/响应处理，无业务逻辑
- [ ] 业务逻辑在 Logic 层实现
- [ ] 错误使用了自定义错误码，而非直接返回
- [ ] 所有日志使用 `WithContext` 保持链路追踪
- [ ] 敏感信息已脱敏处理
- [ ] 配置参数已放入 `etc/*.yaml`，无硬编码
- [ ] 关键业务路径、错误分支和回归风险点已覆盖
- [ ] 事务操作使用项目当前统一的事务封装（如 `TransactCtx`）
- [ ] API/Proto 定义文件与实际代码同步
- [ ] 中间件正确处理了错误和 panic
- [ ] goctl 生成代码与手写业务代码边界清晰

## 审查重点

- 分层职责是否清晰，是否出现 Handler/Model 越界。
- 错误是否有稳定语义，是否保留错误链。
- 事务、缓存、下游调用是否在 Logic 层合理组织。
- 生成代码是否被不必要地直接修改。
