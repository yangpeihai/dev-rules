# 代码审查规范

## 概述

代码审查（Code Review）是保障代码质量、促进知识共享、提升团队整体技术水平的关键环节。本规范涵盖审查流程、角色职责、关注点、反馈规范和效率要求。

## 提交前自检清单（CR-001）

每位开发者在提交代码审查前，必须完成以下自检项：

| 序号 | 检查项 | 验证方式 |
|:---:|:---|:---|
| 1 | 代码能够本地编译/构建通过 | 执行本地构建命令 |
| 2 | 单元测试全部通过 | 运行测试命令 |
| 3 | 静态代码检查无严重错误 | 运行ESLint/Checkstyle等 |
| 4 | 代码格式化符合规范 | 运行Prettier/gofmt等 |
| 5 | 提交信息符合规范 | 参考Git提交规范 |
| 6 | 关联的需求/缺陷单号已标注 | 在PR描述中注明 |
| 7 | 敏感信息已移除 | 人工检查+工具扫描 |
| 8 | 调试代码已清理 | 全局搜索检查 |

## CI/CD前置检查（CR-002）

代码审查请求必须满足以下条件方可提交：
- 代码已推送到远程分支
- PR/MR描述已完整填写
- 所有CI流水线检查通过（构建、单元测试、覆盖率、安全扫描）

**CI检查未通过的PR/MR，Reviewer有权直接驳回**（CR-003）。

## Review角色与职责

### 角色定义

| 角色 | 职责 |
|:---|:---|
| Author | 代码变更的提交者，负责解释代码意图、响应反馈、修复问题 |
| Reviewer | 对代码进行技术审查，提出改进建议，确认代码质量 |
| Approver | 对代码变更进行最终批准，通常由资深工程师或模块负责人担任 |

### 角色分配原则（CR-007）

| 场景 | Reviewer数量 | Approver要求 |
|:---|:---:|:---|
| 常规功能开发 | ≥ 2人 | 至少1名资深工程师 |
| 核心模块变更 | ≥ 2人 | 模块Owner必须参与 |
| 跨团队协作 | ≥ 2人 | 双方Approver共同审批 |
| 紧急Hotfix | ≥ 1人 | 可适当放宽，事后复盘 |
| 大型重构 | ≥ 3人 | 技术负责人必须审批 |

## 审查关注点（CR-008）

### 六维度审查模型

| 维度 | 说明 |
|------|------|
| 功能正确性 | 业务逻辑、边界条件、并发安全、错误处理 |
| 代码风格 | 命名规范、格式、注释、代码组织 |
| 安全性 | 注入攻击、XSS、敏感数据、权限控制 |
| 性能 | 算法复杂度、数据库查询、资源使用、缓存策略 |
| 可维护性 | 单一职责、代码重复、耦合度、可测试性 |
| 测试覆盖 | 单元测试、测试质量、测试命名、测试隔离 |

### 功能正确性检查（CR-009）

| 检查项 | 说明 |
|:---|:---|
| 业务逻辑正确 | 代码实现与需求文档一致 |
| 边界条件处理 | 空值、越界、极值等场景 |
| 并发安全 | 多线程/多进程环境下的正确性 |
| 错误处理 | 异常场景的处理逻辑 |
| 数据一致性 | 数据库事务、缓存一致性 |

### 安全审查（CR-011）

| 风险类别 | 检查要点 |
|:---|:---|
| 注入攻击 | SQL/NoSQL/LDAP/命令注入 |
| XSS攻击 | 输出编码、输入验证 |
| 敏感数据 | 加密存储、传输安全 |
| 权限控制 | 身份认证、授权校验 |

## 评论分级标准（CR-015）

| 级别 | 标识 | 含义 | Author响应要求 |
|:---|:---:|:---|:---|
| **Must Fix** | 🔴 `[MUST]` | 必须修复的问题，阻塞合并 | 必须修复并回复 |
| **Should Fix** | 🟡 `[SHOULD]` | 建议修复的问题 | 建议修复，可协商 |
| **Nice to Have** | 🟢 `[NICE]` | 可选优化建议 | 可选处理 |
| **Question** | 🔵 `[Q]` | 疑问或需要澄清的地方 | 必须回复解释 |

### 评论格式规范（CR-016）

```markdown
[级别标签] 问题描述

详细说明（可选）：
- 为什么这是一个问题
- 可能带来的风险或影响

建议方案（可选）：
```代码示例```

相关链接（可选）：
- 设计文档/需求链接
```

## Review效率要求

### 单次Review代码量限制（CR-018）

| 场景 | 建议最大行数 |
|:---|:---:|
| 常规Review | ≤ 400行 |
| 复杂逻辑变更 | ≤ 200行 |
| 配置/文档变更 | ≤ 800行 |

**当变更超过400行时**，应采取：拆分PR、分阶段Review或线下Review策略（CR-019）。

### Review响应时间要求（CR-020）

| 阶段 | 时间要求 |
|:---|:---:|
| 首次响应 | ≤ 24小时 |
| 紧急修复 | ≤ 4小时 |
| 修改后复验 | ≤ 8小时 |

## 合并前检查清单（CR-025）

```
□ 1. 至少2个Approve（含1个Approver的审批）
□ 2. 所有[MUST]级别评论已解决并得到确认
□ 3. 所有CI/CD流水线检查通过
□ 4. 代码已rebase到目标分支最新状态，无冲突
□ 5. 代码覆盖率满足团队阈值要求
□ 6. 无未解决的安全扫描高危/严重漏洞
□ 7. PR/MR描述完整，包含变更摘要和测试方案
□ 8. 关联的需求/缺陷单号已正确关联
```

## 合并策略（CR-027）

| 策略 | 适用场景 |
|:---|:---|
| **Squash and Merge** | 功能开发、缺陷修复 - 保持主干历史整洁 |
| **Rebase and Merge** | 个人分支、文档更新 - 保持线性历史 |
| **Merge Commit** | 大型特性分支、版本发布 - 保留完整分支历史 |

## 紧急Hotfix流程（CR-029）

```
1. 快速审查（Fast-Track Review）
   - 至少1名Reviewer快速审查
   - 重点关注：问题修复正确性、无副作用

2. 事后补审（Post-Mortem Review）
   - Hotfix发布后24小时内完成正式Review

3. 复盘分析（Root Cause Analysis）
   - 分析缺陷引入原因
   - 制定预防措施
```

## 建设性反馈原则（CR-032）

| 原则 | 说明 |
|:---|:---|
| **对事不对人** | 评论代码而非作者 |
| **解释原因** | 说明为什么需要修改 |
| **提供方案** | 给出具体的改进建议 |
| **认可优点** | 对好的实践给予肯定 |
| **保持谦逊** | 承认自己的理解可能有偏差 |

### 避免负面表达

| ❌ 避免使用 | ✅ 建议使用 |
|:---|:---|
| "你怎么能这样写？" | "这里可能有更好的实现方式..." |
| "这明显是错的" | "这里可能存在XX问题，建议..." |
| "你为什么不..." | "建议考虑...，因为..." |

## 自动化检查工具

### 各语言工具推荐（CR-023）

| 语言 | 工具类型 | 推荐工具 |
|:---|:---|:---|
| Java | 代码风格 | Checkstyle / Spotless |
| Java | 静态分析 | SonarQube / PMD / SpotBugs |
| Java | 安全扫描 | OWASP Dependency-Check |
| JavaScript | 代码风格 | ESLint + Prettier |
| Python | 代码风格 | Black / isort / flake8 |
| Go | 静态分析 | golangci-lint |

### 质量门禁配置（CR-024）

| 门禁项 | 阻断级别 | 阈值建议 |
|:---|:---:|:---|
| 编译/构建失败 | 阻断 | 0错误 |
| 单元测试失败 | 阻断 | 0失败 |
| 代码覆盖率 | 阻断 | 行覆盖率≥80% |
| 高危安全漏洞 | 阻断 | 0高危 |
