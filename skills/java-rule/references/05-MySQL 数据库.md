# MySQL 数据库

## (一) 建表规约

1. **【强制】** 表达是与否概念的字段，必须使用 `is_xxx`的方式命名，数据类型是`unsigned`、`tinyint`（1 表示是，0 表示否）。
   - 说明：任何字段如果为非负数，必须是`unsigned`。
   - 正例：表达逻辑删除的字段名`is_deleted`，1 表示删除，0 表示未删除。

2. **【强制】** 表名、字段名必须使用小写字母或数字，禁止出现数字开头，禁止两个下划线中间只出现数字。数据库字段名的修改代价很大，因为无法进行预发布。
   - 正例：`aliyun_admin / rdc_config / level3_name`
   - 反例：`AliyunAdmin / rdcConfig / level_3_name`

3. **【强制】** 表名不使用复数名词。
   - 说明：表名应该仅仅表示表里面的实体内容，不应该表示实体数量，对应于 DO 类名也是单数形式，符合 DDD 的简洁思想。

4. **【强制】** 禁用保留字，如 desc、range、match、delayed 等，请参考 MySQL 官方保留字。

5. **【强制】** 主键索引名为`pk_字段名`；唯一索引名为`uk_字段名`；普通索引名则为`idx_字段名`。
   - 说明：pk_ 即 primary key；uk_ 即 unique key；idx_ 即 index 的简称。

6. **【强制】** 小数类型为`decimal`，禁止使用 `float`和`double`。
   - 说明：float 和 double 在存储的时候，存在精度损失的问题，很可能在值的比较时，得到不正确的结果。如果存储的数据范围超过 decimal 的范围，建议将数据拆成整数和小数分开存储。

7. **【强制】** 如果存储的字符串长度几乎相等，使用 `char`定长字符串类型。

8. **【强制】** `varchar`是可变长字符串，不预先分配存储空间，长度不要超过 5000，如果存储长度大于此值，定义字段类型为`text`，独立出来一张表，用主键来对应，避免影响其它字段索引效率。

9. **【强制】** 表必备三字段：`id`, `gmt_create`, `gmt_modified`。
   - 说明：其中 `id` 必为主键，类型为`bigint unsigned`、单表时自增、步长为 1。`gmt_create` 和`gmt_modified` 的类型均为：`datetime` 类型，`gmt_create` 默认值为`NOW()`，`gmt_modified` 默认值为`CURRENT_TIMESTAMP()`。

10. **【推荐】** 表的命名最好是加上"业务名称_表的作用"。
    - 正例：`alipay_task / force_project / trade_config`

11. **【推荐】** 库名与应用名称尽量一致。

12. **【推荐】** 修改字段类型时，必须保证该字段在其它 SQL 下是兼容的。

13. **【推荐】** 如果修改字段含义或对字段表示的状态追加时，记得第一时间更新该字段的相关文档。

14. **【推荐】** 表设计时注意：
    - 1）表的字段设计要考虑扩展性，新增字段时，确保现有代码的兼容性
    - 2）对字段进行注释说明
    - 3）设计文档与数据库设计保持同步更新

15. **【参考】** 字段允许适当冗余，以提高查询性能，但必须考虑数据一致。冗余字段应遵循：
    - 1）不是频繁修改的字段
    - 2）不是 varchar 超长字段，更不能是 text 字段
    - 正例：商品类目名称使用频率高，字段长度短，名称基本一成不变，可在表结构中冗余存储，避免关联查询

16. **【参考】** 利用数据分区技术，可以按业务需求将数据水平或垂直分区，提升查询性能。

## (二) 索引规约

1. **【强制】** 业务上具有唯一特性的字段，即使是多个字段的组合，也必须建成唯一索引。
   - 说明：不要以为唯一索引影响了 insert 速度，这个速度损耗可以忽略，但提高查找速度是明显的；另外，即使在应用层做了非常完善的校验控制，但只要没有唯一索引，根据墨菲定律，必然有脏数据产生。

2. **【强制】** 超过三个表禁止 join。需要 join 的字段，数据类型必须绝对一致；多表关联查询时，保证被关联的字段需要有索引。
   - 说明：即使双表 join 也要注意表索引、SQL 性能

3. **【强制】** 在 varchar 字段上建立索引时，必须指定索引长度，没必要对全字段建立索引，根据实际文本区分度决定索引长度即可。
   - 说明：索引的长度与区分度是一对矛盾体，一般对字符串类型索引，设置为 10~20 的索引长度，区分度就已经很高了。

4. **【强制】** 页面搜索严禁左模糊或者全模糊，如果需要请走搜索引擎来解决。
   - 说明：索引文件具有 B+Tree 的最左前缀匹配特性，如果左边的值未确定，那么无法使用此索引。

5. **【推荐】** 注意区分覆盖索引和回表查询，避免不必要的回表。
   - 说明：覆盖索引是指查询的列正好是索引的列，不需要回表查询。

6. **【推荐】** 利用覆盖索引来进行查询操作，避免回表。
   - 说明：覆盖索引是指查询的列正好是索引的列，不需要回表查询。

7. **【推荐】** 利用延迟关联或者子查询优化超多分页场景。
   - 说明：MySQL 并不是跳过 offset 行，而是取 offset+N 行，然后返回放弃前 offset 行，返回 N 行。可以通过子查询先查出主键，再关联主表，减少排序和扫描的数据量。

8. **【推荐】** SQL 性能优化的目标：至少要达到 range 级别，要求是 ref 级别，如果可以是 consts 最好。
   - 说明：
     - consts：单表中最多只有一个匹配行（主键或者唯一索引），在优化阶段即可读取到数据
     - ref：指的是使用普通的索引（normal index）
     - range：对索引进行范围检索

9. **【推荐】** 慢查询的排查与优化：
   - 1）使用慢查询日志记录慢 SQL
   - 2）使用 explain 分析 SQL 执行计划
   - 3）根据执行计划优化 SQL 或索引

10. **【参考】** 索引并不是越多越好，索引太多会占用额外的存储空间，影响 insert、update 的速度。

11. **【参考】** 建索引的原则：
    - 1）搜索的索引列，不一定是所要选择的列
    - 2）字符字段建立索引要有长度限制
    - 3）区分度不高的字段不适合建立索引

## (三) SQL 语句

1. **【强制】** 不要使用 COUNT (列名) 或 COUNT (常量) 来替代 COUNT (*)，COUNT (*) 是 SQL92 定义的标准统计行数的语法，跟数据库无关，跟 NULL 和非 NULL 无关。
   - 说明：COUNT (*) 会统计值为 NULL 的行，而 COUNT (列名) 不会统计此列为 NULL 值的行。

2. **【强制】** COUNT (DISTINCT col) 计算该列除 NULL 之外的不重复行数，注意 COUNT (DISTINCT col1, col2) 如果其中一列全为 NULL，那么即使另一列有不同的值，最后结果也是 0。

3. **【强制】** 当某一列的值全是 NULL 时，COUNT (col) 的返回结果为 0，但 SUM (col) 的返回结果为 NULL，因此使用 SUM () 时需注意 NPE 问题。
   - 正例：使用以下方法来避免 NPE 问题：
     ```java
     Integer count = dao.count(query);
     return count == null ? 0 : count;
     ```

4. **【强制】** 使用预编译语句进行数据库操作，避免 SQL 注入。
   - 正例：
     ```java
     PreparedStatement ps = conn.prepareStatement("SELECT * FROM users WHERE id = ?");
     ps.setLong(1, id);
     ```

5. **【强制】** 禁止使用存储过程，存储过程难以调试和扩展，更没有移植性。

6. **【强制】** `iBATIS`使用`#variable#`方式，可以防止 SQL 注入；`MyBatis`使用`#{variable}` 方式。

7. **【推荐】** 代码中写分页查询逻辑时，若 count 结果为 0 应该直接 return，不要执行后续查询逻辑。

8. **【推荐】** 不得使用外键与级联，一切外键概念必须在应用层解决。
   - 说明：学生成绩中的 student 与 student 编号是主从关系，只要 student 编号存在，那么成绩记录就存在，如果删掉学生记录，那么对应的成绩记录自然也需要删掉，但是这种级联关系，不通过数据库的外键机制来约束，而是在应用层进行代码控制。

9. **【推荐】** 禁止使用存储过程。

10. **【参考】** 不要使用一个大事务，将不相关的数据库操作放在一起，应该将大事务拆分成多个小事务。

## (四) ORM 映射

1. **【强制】** 表名与类名、字段名与属性名映射一致。

2. **【强制】** 在表查询中，一律不要使用 * 作为查询的字段列表，需要哪些字段必须明确写出来。
   - 说明：
     - 1）增加查询解析代价，消耗更多网络带宽
     - 2）可能拿到不需要的敏感数据，存在安全隐患
     - 3）如果字段类型变更，可能导致程序错误

3. **【推荐】** 使用 MyBatis 等 ORM 框架时，使用 ResultMap 明确字段映射关系。

4. **【推荐】** POJO 类属性使用包装类型，RPC 方法的返回值和参数使用包装类型。

5. **【参考】** 配置 MyBatis 的缓存机制，减少数据库访问。
