# 安全规约

1. **【强制】** 隶属于用户个人的页面或者功能必须进行权限控制校验。
   - 说明：防止没有做水平权限校验就可随意访问、修改、删除别人的数据，比如查看他人的私信内容、修改他人的订单。

2. **【强制】** 用户敏感数据禁止直接展示，必须对展示数据进行脱敏。
   - 说明：个人手机号码显示为:158****9119，隐藏中间 4 位，防止隐私泄露。
   - 常见脱敏场景：
     - 手机号：隐藏中间 4 位
     - 身份证号：隐藏出生日期部分
     - 银行卡号：仅显示后 4 位
     - 邮箱：隐藏@前的部分字符

3. **【强制】** 用户输入的 SQL 参数严格使用参数绑定或者 METADATA 字段值限定，防止 SQL 注入，禁止字符串拼接 SQL 访问数据库。
   - 正例：使用 PreparedStatement
   - 反例：`String sql = "SELECT * FROM users WHERE name = '" + userName + "'";`

4. **【强制】** 用户请求传入的任何参数必须做有效性验证。
   - 说明：忽略参数校验可能导致：
     - page size 过大导致内存溢出
     - 恶意 order by 导致数据库慢查询
     - 任意重定向
     - SQL 注入
     - 反序列化注入
     - 正则输入源串拒绝服务 ReDoS
   - 说明：Java 代码用正则来验证客户端的输入，有些正则写法验证普通用户输入没有问题，但是如果攻击人员使用的是特殊构造的字符串来验证，有可能导致死循环的结果。

5. **【强制】** 禁止向 HTML 页面输出未经安全过滤或未正确转义的用户数据。
   - 说明：防止 XSS（跨站脚本攻击）

6. **【强制】** 表单、AJAX 提交必须执行 CSRF（Cross-site request forgery）安全过滤。
   - 说明：CSRF 跨站请求伪造是一类常见编程漏洞。对于存在 CSRF 漏洞的应用/网站，攻击者可以事先构造好 URL，只要受害者用户一访问，后台便在用户不知情情况下对数据库中用户参数进行相应修改。
   - 防护措施：
     - 使用 CSRF Token
     - 验证 Referer 头
     - 使用验证码

7. **【强制】** 在使用平台资源，譬如短信、邮件、电话、下单、支付，必须实现正确的防重放限制，如数量限制、疲劳度控制、验证码校验，避免被滥刷导致资损。
   - 说明：如注册时发送验证码到手机，如果没有限制次数和频率，那么可以利用此功能骚扰到其它用户，并造成短信平台资源浪费。

8. **【推荐】** 发贴、评论、发送即时消息等用户生成内容的场景必须实现防刷、文本内容违禁词过滤等风控策略。
   - 说明：防止恶意用户发布违法、违规内容

## 常见安全漏洞及防护

### SQL 注入
- **原因**：用户输入被直接拼接到 SQL 语句中
- **防护**：使用参数绑定（PreparedStatement）、使用 ORM 框架

### XSS 攻击
- **原因**：用户输入被直接输出到 HTML 页面
- **防护**：输出转义、使用安全框架的转义工具

### CSRF 攻击
- **原因**：利用用户已登录的身份执行恶意请求
- **防护**：CSRF Token、验证 Referer

### 越权访问
- **原因**：未进行权限校验或权限校验不完整
- **防护**：所有涉及用户数据的操作都必须进行权限校验

### 敏感信息泄露
- **原因**：日志、异常信息中包含敏感数据
- **防护**：日志脱敏、异常信息过滤
