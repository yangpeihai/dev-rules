# Go 代码检查助手

帮助 AI 或开发者在已确认的检查范围内审查 Go 代码，优先识别当前变更或目标模块中的高价值问题，并输出可供人工处理或 `code-fixer-go` 继续消费的结构化结果。

## Overview

这个 skill 的重点不是“多报一些问题”，而是“在正确范围内，按正确规则，找出真正值得修的问题”。

它默认服务于两类场景：
- 单轮快速 review：直接给出高价值 findings 和简短结论
- 自检流水线：输出可被 `code-fixer-go` 继续消费的结构化问题项

## AI 速用入口

如果只看这一节，也应按下面执行：

1. 先确认检查范围，不擅自扩大
2. 再判断项目类型：通用 Go、Gin Web、go-zero
3. 只加载最相关的规则：优先项目规则，其次 `go-rule`，再补框架规则
4. 先查高风险问题：安全、错误处理、事务/资源/并发、分层越界
5. 输出时先列 findings，再给简短总结
6. 如果后续可能要修，优先输出统一字段：`issue_id`、`severity`、`location`、`summary`、`basis`、`suggested_fix`、`status`

### 最短输出模板

```markdown
## Findings
- [GO-H001][high][confirmed] [文件:行号] 问题摘要
  影响：
  依据：
  修复方向：

## Summary
- 可直接修复：
- 需进一步验证：
```

### 速停条件

出现以下情况时，不要继续扩大结论：
- 范围不清
- 项目类型判断不出来，且会影响结论
- 缺少关键上下文
- 该问题本质上需要运行验证

## AI 触发条件

当任务满足以下任一条件时，应优先使用本规范：
- 审查 Go 代码质量、工程规范或架构边界
- 对指定文件、目录、模块、提交做定向检查
- 需要产出可追踪的 findings 报告
- 在修复前先做 AI 自检，确认问题列表和修复优先级

## AI 决策顺序

按以下顺序处理，避免一上来就机械扫描：

1. 先确认检查范围：文件、目录、模块、提交，还是全项目
2. 再确认检查目标：快速 review、结构化报告、修复前自检
3. 再判断项目类型：通用 Go、Gin Web、go-zero
4. 再选择规则来源：用户要求 > 项目规则 > 当前 skill 约定 > 通用 Go 规范
5. 再缩小检查维度：先高风险，再一致性
6. 最后决定输出模式：快速模式或结构化模式

## 核心边界

### 必须遵守

- 只检查，不修改源代码
- 只在用户指定范围内检查，除非用户明确要求扩大范围
- 结论必须基于代码事实、规则依据或明确推断
- 优先报告真实行为风险，不用低价值风格噪音淹没结果

### 不应做的事

- 不把个人习惯包装成“项目规范”
- 不在证据不足时断言性能、并发或覆盖率一定有问题
- 不把历史项目直接按理想新项目标准生硬判死
- 不为了凑整齐格式而牺牲信息密度

## 检查范围确定

在开始检查前，必须先明确本轮扫描范围，不得默认扩大。

### 范围优先级

1. 用户显式指定的范围
2. 当前任务直接相关的变更范围
3. 为验证结论所必需的最小关联范围
4. 仅在用户明确要求时，才执行全盘扫描

### 常见范围类型

| 范围类型 | 说明 | 适用场景 |
|------|------|------|
| 本次变更扫描 | 仅检查当前提交、当前分支 diff 或指定变更文件 | PR review、提交前自检、快速审查 |
| 定向模块扫描 | 检查某个目录、模块、服务或子系统 | 局部重构、专项治理 |
| 提交级扫描 | 以某几个 commit 为边界审查 | 回溯问题、审查历史变更 |
| 全盘扫描 | 检查整个仓库或整个 Go 服务 | 专项体检、基线治理、全面审查 |

### 默认策略

- 用户未明确说明时，默认优先按“本次变更扫描”处理
- 如果仅扫描变更范围不足以判断风险，可扩展到最小必要上下文
- 扩展范围时，应明确说明为什么需要扩展、扩展到了哪里
- 未经用户明确授权，不默认做全盘扫描

## 规则路由

默认按以下顺序加载和解释规则：

1. 用户本轮显式指令
2. 当前项目中的规则文档、skills、团队约定
3. `go-rule`
4. `gin-web-rule` 或 `go-zero-rule`
5. Go 通用惯例与标准工具链

### 项目类型判断

| 项目类型 | 典型信号 | 优先联动规范 |
|------|------|------|
| Gin Web | 使用 `gin`、存在 `handler -> logic -> repository`、常见 `gorm`/统一返回体 | `gin-web-rule` + `go-rule` |
| go-zero | 存在 `.api`/`.proto`、`ServiceContext`、`logic`、`goctl` 痕迹 | `go-zero-rule` + `go-rule` |
| 通用 Go | 无明显框架分层特征 | `go-rule` |

### 路由原则

- 通用问题，如错误处理、`context`、资源释放、并发正确性，默认由 `go-rule` 兜底
- 框架边界问题，如 Gin 的 Handler/Logic/Repository、go-zero 的 `handler/server -> logic -> model`，优先由框架规则判断
- 同一问题同时涉及通用 Go 与框架分层时，先用框架规则判断落点，再用 `go-rule` 判断写法是否合理

## 默认检查维度

默认按以下维度检查，可根据任务收缩：

| 维度 | 重点关注 |
|------|------|
| 分层与架构 | 目录边界、职责划分、依赖方向 |
| 错误处理 | `err` 检查、错误链、错误映射 |
| 资源与并发 | `context`、超时、资源释放、竞态风险 |
| 数据访问与事务 | 参数化查询、事务边界、大表操作、缓存边界 |
| 安全 | 输入校验、密钥、日志脱敏、敏感信息 |
| API 与契约 | 命名、签名、返回体、兼容性 |
| 测试与验证 | 关键路径、错误分支、回归风险点 |
| 一致性与可维护性 | 命名、格式化、项目既有约定 |

## 检查顺序

默认按以下顺序收敛 findings：

1. 安全问题
2. 行为错误和错误处理缺陷
3. 事务边界、资源泄漏、并发风险
4. 框架分层越界和架构问题
5. 测试缺口与可验证性问题
6. 一致性和可维护性问题

只有在前面的高价值项基本明确后，才继续看低风险一致性问题。

## 问题分级

| 字段 | 含义 |
|------|------|
| `severity` | `high` / `medium` / `low` |
| `confidence` | `high` / `medium` / `low` |
| `status` | `confirmed` / `suspected` / `needs_verification` |

### 判定要求

- `confirmed`：有足够代码证据，问题可以直接成立
- `suspected`：有较强迹象，但仍有上下文缺口
- `needs_verification`：需要运行、压测、联调或更多上下文后才能下结论

不要把需要验证的问题伪装成确定问题。

## 推荐 findings 字段

为了兼容单轮输出和后续修复，推荐每个问题项尽量包含以下字段：

| 字段 | 说明 |
|------|------|
| `issue_id` | 问题编号，如 `GO-H001` |
| `severity` | 严重级别 |
| `category` | 问题类别 |
| `scope` | 文件、目录、模块或提交范围 |
| `location` | 文件和行号，无法精确定位时写结构位置 |
| `summary` | 一句话问题描述 |
| `impact` | 对行为、安全、维护性的影响 |
| `basis` | 依据的规则或代码事实 |
| `suggested_fix` | 修复方向，保持最小必要 |
| `confidence` | 结论置信度 |
| `status` | `confirmed` / `suspected` / `needs_verification` |

## Issue ID 建议

- 高风险问题：`GO-H001`、`GO-H002`
- 中风险问题：`GO-M001`、`GO-M002`
- 低风险问题：`GO-L001`、`GO-L002`

如果用户已有问题编号，优先沿用用户编号。

## 默认评分规则

`code-checker-go` 默认应给出评分，但这个评分不是“整个项目质量总分”，而是“本轮检查目标范围内代码质量评分”。

### 评分边界

- 只评本轮检查目标范围
- 不擅自扩大到整个项目、整个仓库或无关模块
- 评分应结合问题严重度、问题数量、证据充分性和残余风险综合判断
- 评分必须和本轮 findings 一致，不能为了好看虚高

### 默认输出项

默认应包含以下评分字段：

- `score`：0-100 分
- `grade`：等级，如 `A` / `B` / `C` / `D`
- `scoring_basis`：评分依据，说明为什么得到这个分数

### 评分参考

评分时可参考以下因素：

- `high` / `medium` / `low` 问题的数量和分布
- 是否存在 `confirmed` 问题
- 是否仍有较多 `needs_verification` 或 `suspected`
- 问题是否集中在安全、错误处理、事务、并发、分层等高风险区域
- 目标范围内是否存在明显架构或行为风险

### 评分表达要求

- 评分必须在检查结果总结中明确给出
- 必须同时给出分数、等级和评分依据
- 如果证据不足，应在 `scoring_basis` 中明确说明
- 不得把“问题未验证”包装成“高质量”

## 双模式输出

### 快速 Review 模式

适用场景：
- 用户只要高价值问题
- 只检查少量文件
- 本轮不一定继续自动修复

输出要求：
- 先列 findings
- 每个问题包含位置、影响、依据、修复方向
- 最后给 1 段简短总结
- 默认同步生成 Markdown 文档，并在可行时同步生成对应 JSON 文件
- 默认给出目标范围代码质量评分：`score`、`grade`、`scoring_basis`

### 结构化报告模式

适用场景：
- 需要给 `code-fixer-go` 继续消费
- 需要多轮跟踪问题
- 需要留存审查报告

输出要求：
- 保留统一字段
- 明确问题状态与置信度
- 标出可直接修复项与需验证项
- 默认落盘为 JSON，并同时补一份 Markdown 摘要
- 默认给出目标范围代码质量评分：`score`、`grade`、`scoring_basis`

## 输出落盘规则

默认应将检查结果保存为文档，而不是只停留在对话输出中。

### 默认目录

- 默认输出目录：`docs/code-review/`
- 如果目录不存在，应先创建目录，再写入结果文件
- 除非用户明确指定其他位置，否则不要把检查结果直接散落到 `docs/` 根目录

### 默认文件格式

- 默认同时输出两份文件：
- 一份 Markdown：给人快速阅读和 review
- 一份 JSON：给后续 AI、脚本或修复流程继续消费

### 文件命名规则

默认文件名应包含以下信息：

1. 日期
2. 时间
3. 用户本轮任务主题
4. 从当前分支名提取的简短特性名

推荐格式：

- Markdown：`YYYY-MM-DD-HHMMSS-<task-topic>-<branch-feature>.md`
- JSON：`YYYY-MM-DD-HHMMSS-<task-topic>-<branch-feature>.json`

### 命名生成原则

- `task-topic` 优先使用用户本轮任务主题，转成简短、可读、短横线连接的英文或拼音标识
- `branch-feature` 作为补充信息，从当前分支名中提取更短的特性名
- 如果分支名本身已经较短且清晰，可直接使用
- 如果分支名包含噪音前缀、用户名、流水号等信息，应只保留最能表达特性的那一段

### 落盘要求

- 对话中给出 findings 摘要时，也应同步说明写入了哪些文件
- Markdown 与 JSON 的核心结论必须保持一致
- 文件中必须包含检查范围、检查时间、规则来源、Findings、结论和建议下一步

## 与 `code-fixer-go` 的联动示例

如果后续要进入自纠流程，`code-checker-go` 的 findings 至少应让下游拿到“问题是什么、依据是什么、建议怎么改、当前是否能直接修”。

```json
{
  "metadata": {
    "project_type": "go-zero",
    "scope": "internal/handler/user/createuserhandler.go",
    "rules": ["go-zero-rule", "go-rule"],
    "score": 72,
    "grade": "C",
    "scoring_basis": "目标范围内存在已确认的高风险分层问题，且会影响后续错误处理和事务收敛"
  },
  "findings": [
    {
      "issue_id": "GO-H001",
      "severity": "high",
      "category": "layering",
      "location": "internal/handler/user/createuserhandler.go:28",
      "summary": "Handler 中直接编排数据库写入，绕过 logic 层",
      "impact": "破坏 go-zero 分层边界，后续错误处理和事务收敛会变得困难",
      "basis": "go-zero-rule 要求 handler/server -> logic -> model 分层",
      "suggested_fix": "将业务编排迁移到对应 logic，由 handler 仅做请求解析和响应输出",
      "confidence": "high",
      "status": "confirmed"
    }
  ]
}
```

这里的关键不是 JSON 形式本身，而是字段语义要完整，足够让 `code-fixer-go` 逐项消费。

## AI 默认执行步骤

1. 明确检查范围和输出目标，不擅自扩大任务
2. 判断项目类型，并加载最相关的 1-2 套规则
3. 先查高风险问题，再查一致性问题
4. 每个 findings 明确区分确定问题、合理怀疑、需验证事项
5. 如果后续可能要修复，尽量输出统一字段和问题编号
6. 总结时优先说明真正影响行为和风险的结论
7. 基于目标范围内代码质量给出 `score`、`grade`、`scoring_basis`
8. 将结果同时整理为 Markdown 和 JSON，并默认写入 `docs/code-review/`

## 基础验证清单

当项目上下文、工具链和用户授权允许时，建议在代码审查之外补做最小必要的运行验证。它的作用不是替代审查，而是帮助区分“已确认问题”和“仍需运行验证的问题”。

| 序号 | 检查项 | 验证方式 |
|:---:|:---|:---|
| 1 | 代码能够本地编译或构建通过 | 执行 `go build ./...` 或项目约定构建命令 |
| 2 | 单元测试全部通过 | 执行 `go test ./...` 或目标模块测试命令 |
| 3 | 静态检查无严重错误 | 执行 `golangci-lint run`、`staticcheck ./...` 或项目约定检查命令 |
| 4 | 代码格式符合规范 | 执行 `gofmt -w`、`gofumpt -w` 或项目约定格式检查命令 |

### 使用原则

- 只在用户允许的范围内运行验证，不擅自扩大到全仓库或无关模块
- 如果项目未接入某项工具，应明确说明缺口，不要假装已经完成验证
- 如果无法运行验证，应将相关结论标记为 `needs_verification`，避免过度下结论
- 运行验证结果应体现在最终报告中，与 findings、评分和建议下一步保持一致

## AI 自检清单

输出前至少快速检查：

- [ ] 是否只检查了用户允许的范围
- [ ] 是否优先使用了项目规则，再回退到通用规则
- [ ] 是否先报高价值问题，再报低价值问题
- [ ] 是否区分了 `confirmed`、`suspected`、`needs_verification`
- [ ] 是否给出了足够明确的修复方向，而不是空泛建议
- [ ] 是否能让 `code-fixer-go` 直接消费这些 findings
- [ ] 是否已给出目标范围代码质量评分
- [ ] 是否同时包含 `score`、`grade`、`scoring_basis`
- [ ] 是否评分与 findings 严重度、证据和残余风险相匹配
- [ ] 是否已明确说明哪些基础验证已执行、哪些未执行及原因
- [ ] 是否已将结果写入默认文档目录
- [ ] 是否文件名包含日期时间、任务主题和分支特性名
- [ ] 是否同时生成了 Markdown 和 JSON 两份结果

## 停止条件

出现以下情况时，应停止继续扩大结论：

- 范围不清，继续检查会超出用户授权
- 项目类型无法判断，且规则差异会影响结论
- 缺少关键上下文，继续下结论风险过高
- 某些问题必须依赖运行结果、联调结果或更多代码上下文

此时应明确说明缺口，而不是继续猜。

## AI 默认输出要求

- 默认先输出 findings，再给简短总结
- findings 按严重程度排序，高风险优先
- 每个问题尽量包含位置、影响、依据、修复方向
- 用户未要求评分时，不强行给总分
- 默认给出目标范围代码质量评分：`score`、`grade`、`scoring_basis`
- 默认将结果写入 `docs/code-review/`

## 何时联动其他文档

- **通用 Go 审查** → `go-rule`
- **Gin Web 分层、事务、返回体审查** → `gin-web-rule` + `go-rule`
- **go-zero 分层、错误处理、ServiceContext 审查** → `go-zero-rule` + `go-rule`
- **只做快速 review** → 当前 skill + 最相关的一套规则

## 常见误区

| 误区 | 正确做法 |
|------|------|
| 只看格式和命名 | 先看行为风险、边界和错误处理 |
| 把所有怀疑都当成确定问题 | 用 `status` 和 `confidence` 标明不确定性 |
| 框架项目只套通用 Go 规范 | 先用框架规则判断职责边界 |
| 为了完整而扫太大范围 | 只看用户允许的范围，必要时说明缺口 |

## 最终目标

这个 skill 的产出应当满足两件事：

1. 人可以快速看懂哪里真有风险
2. AI 可以直接基于 findings 进入 `code-fixer-go` 的修复闭环
