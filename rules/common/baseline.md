# 安全与工程基线

> 核心准则：默认安全，质量内建。适用于所有语言和框架。

## 1. 安全基线 (Security Baseline)

### 输入/输出处理
- ✅ **输入验证**：所有外部输入必须经过白名单验证或强类型校验
- ✅ **防 SQL 注入**：数据库操作必须使用参数化查询或成熟 ORM
- ✅ **防 XSS**：所有输出到前端的数据必须进行上下文编码

### 认证与授权
- ✅ **密码存储**：强制使用 `bcrypt` / `argon2` 哈希加盐
- ✅ **Token 策略**：JWT 有效期 ≤ 15分钟，包含合理的 refresh 机制
- ✅ **会话管理**：空闲超时 30 分钟，绝对超时 24 小时
- ✅ **Cookie 安全**：必须设置 `HttpOnly`, `Secure`, `SameSite`
- ✅ **权限控制**：实施基于 RBAC 的最小权限原则，操作需校验资源归属

### 数据与日志保护
- ✅ **密钥管理**：禁止硬编码，必须从环境变量或 KMS 读取
- ✅ **日志脱敏**：禁止记录密码、Token、明文手机号/邮箱
- ✅ **错误屏蔽**：对外不暴露系统异常堆栈和底层 SQL 信息

### API 与网络安全
- ✅ **防滥用**：敏感接口实施速率限制 (Rate Limiting)
- ✅ **防 CSRF**：状态改变操作必须校验 CSRF Token (如非前后端分离)
- ✅ **防御性超时**：外部服务调用必须设置明确的超时时间

## 2. 代码与架构基线 (Architecture & Code)

### 架构原则
- **分层设计**：严格遵循 `API → Service → Repository` 单向依赖
- **高内聚低耦合**：单一职责原则，依赖抽象(接口)而非具体实现
- **防御性编程**：不信任外部依赖，妥善处理空值(Null)和异常分支

### 测试要求
- **单元测试**：新核心逻辑必须覆盖单元测试，覆盖率目标 ≥ 80%
- **测试独立性**：遵循 AAA 模式 (Arrange-Act-Assert)，测试间禁止相互依赖
- **外部依赖**：强制使用 Mock 隔离数据库、网络等外部 IO

## 3. 常见标准错误码 (Standard Error Codes)

| HTTP 状态 | 错误码 | 说明 |
|:---|:---|:---|
| `400` | 400001 / 400002 | 通用参数错误 / JSON 解析错误 |
| `401` | 401001 / 401002 | Token 过期 / Token 无效 |
| `403` | 403001 | 权限不足 (RBAC拦截) |
| `404` | 404001 | 资源不存在或无权访问 |
| `409` | 409001 | 资源状态冲突 (如重复创建) |
| `429` | 429001 | 请求过多（触发限流） |
| `500` | 500001 | 内部服务器错误 (未捕获异常) |
